# 零成本数据收集网页搭建实战手册

> **项目**：4D天性测评系统  
> **目标**：用纯免费方案搭建一个"用户填写 → 数据回收 → 统计可视化"的完整链路  
> **核心挑战**：微信内置浏览器对 `*.workers.dev` 的拦截  
> **最终架构**：GitHub Pages（前端）+ Cloudflare Worker（API）+ D1（数据库）+ 飞书 Webhook（消息通知/兜底）

---

## 一、项目背景与核心诉求

### 1.1 场景定义
- **用户端**：通过一个网页完成测评/问卷/表单填写
- **数据端**：收集用户填写的数据，存入数据库，可统计、可导出
- **管理端**：有一个后台页面查看数据、做批量操作
- **成本要求**：全程零费用（域名除外，GitHub Pages 自带免费域名）

### 1.2 用户打开场景
本项目需要覆盖两种最常见的用户打开方式：

| 场景 | 占比 | 特点 |
|------|------|------|
| **手机浏览器**（Safari/Chrome/系统浏览器）| ~30% | 网络环境相对自由，fetch 正常 |
| **微信内置浏览器**（公众号/群聊/朋友圈）| ~70% | 微信有严格的域名白名单和拦截策略 |

**核心矛盾**：一套代码要同时满足浏览器自由网络和微信受限网络的数据上报。

## 三、浏览器环境链路（正常路径）

这是"理想情况"下的完整数据流：

```javascript
// 浏览器环境 - 正常 fetch 到 Worker
fetch('https://d4-assessment-api.avenli0.workers.dev/api/submit', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    username: name,
    F: F, N: N, T: T, S: S,
    scores: { green: g, yellow: y, blue: b, orange: o },
    mainColor: d.name + '·' + d.type
  })
})
.then(res => res.json())
.then(data => {
  // Worker 返回 {success: true}
  // 数据已入库，管理后台可查看
});
```

**Worker 端处理**：
```javascript
// Worker: 接收 → 入库 → 返回
async function saveResult(db, data) {
  const source = data.source || 'web';  // 标记数据来源
  await db.prepare(`
    INSERT INTO results (name, f, t, n, s, green, yellow, blue, orange, primary_color, primary_type, source)
    VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?)
  `).bind(...).run();
  return { success: true };
}
```

**特点**：
- ✅ 数据实时入库
- ✅ 有 `source='web'` 标记，便于统计
- ✅ 管理后台可即时查看

## 四、微信环境链路（核心难点）

### 4.1 问题现象

微信内打开网页，点击提交按钮后：
- 页面无响应，提交状态一直显示"正在保存..."
- 飞书消息延迟 1-2 分钟才收到
- 数据库里没有数据

### 4.2 根因分析：微信对 `*.workers.dev` 的拦截

**不是** DNS 解析失败，**不是** 404，**而是 TCP 层面的静默挂起。**

```
浏览器 fetch → 发起 HTTPS 连接到 d4-assessment-api.avenli0.workers.dev
                    ↓
            微信内置浏览器拦截该域名
                    ↓
            TCP SYN 包发出后无响应
                    ↓
            浏览器等待 TCP 超时（默认 75 秒 ~ 2 分钟）
                    ↓
            超时后 fetch catch 触发
                    ↓
             fallback 逻辑执行（发飞书消息）
```

**关键认知**：微信的拦截是**静默的**——不会报错、不会弹窗、不会返回 403，只是让连接"消失在空气中"。这导致 `fetch` 不会立即 `catch`，而是**挂起等待超时**。

### 4.3 微信拦截的范围

| 域名类型 | 是否被拦截 | 备注 |
|----------|-----------|------|
| `*.workers.dev` | ❌ **拦截** | Cloudflare Worker 默认域名 |
| `*.github.io` | ✅ 正常 | GitHub Pages，微信可访问 |
| `*.vercel.app` | ⚠️ 不稳定 | 有时能通，有时被限 |
| `*.netlify.app` | ⚠️ 不稳定 | 同上 |
| 国内备案域名 | ✅ 正常 | 如腾讯云、阿里云服务器 |
| 飞书域名 `open.feishu.cn` | ✅ 正常 | 微信不拦截 |

**结论**：如果前端托管在 GitHub Pages，API 用 Cloudflare Worker 默认域名，微信用户提交时 API 请求会被静默拦截。

### 4.4 运营商层面的额外限制

在实际测试中发现，**不同网络环境差异巨大**：

| 网络环境 | 对 `*.workers.dev` 访问 | 备注 |
|----------|------------------------|------|
| 中国移动 4G/5G | ✅ 基本正常 | 偶有延迟 |
| 中国联通 4G/5G | ❌ **完全阻断** | TCP 无法建立连接 |
| 中国电信 | ⚠️ 不稳定 | 时通时不通 |
| 家庭宽带（电信）| ✅ 正常 | 固网限制较少 |
| 公司/校园网 | ⚠️ 因网而异 | 取决于防火墙策略 |

**这意味着**：即使浏览器环境，如果用联通网络，Worker 也可能连不上。所以飞书兜底不是"微信专属"，而是"全场景保底"。

## 五、解决方案演进（踩坑记录）

### 5.1 方案一：纯 Worker 链路（失败）

**思路**：前端直接 `fetch` Worker，不管环境。

**结果**：
- 浏览器 ✅
- 微信 ❌（TCP 挂起 1-2 分钟）
- 联通网络 ❌

**教训**：默认域名在受限网络下不可行。

### 5.2 方案二：自定义域名 + Worker（可行但需额外配置）

**思路**：给 Worker 绑定一个自己的域名（如 `api.yourdomain.com`），避开 `*.workers.dev`。

**可行性**：
- ✅ 需要有自己的域名（~60元/年）
- ✅ 在 Cloudflare 上配置 DNS 和 Worker Route
- ✅ 微信不再拦截
- ❌ 增加了域名成本和配置复杂度
- ❌ 不是"零成本"方案

**结论**：适合长期运营项目，但不符合本项目"纯免费"目标。

### 5.3 方案三：微信检测 + 直发飞书（最终采用）

**思路**：前端检测当前环境，如果是微信/企业微信，**跳过 Worker，直接发飞书消息**。

```javascript
// 微信环境检测（三保险）
var ua = navigator.userAgent.toLowerCase();
var isWechat = ua.indexOf('micromessenger') !== -1 
            || ua.indexOf('wxwork') !== -1 
            || typeof WeixinJSBridge !== 'undefined';

if (isWechat) {
  // 微信环境：跳过 Worker，直接发飞书
  sendFeishu('需手动导入', '已通知飞书备份');
} else if (hasCF) {
  // 浏览器环境：请求 Worker 入库
  fetch(CF_API + '/api/submit', {...})
    .catch(err => sendFeishu('需手动导入', '云端保存失败'));
} else {
  sendFeishu('', '已通知飞书备份');
}
```

**为什么飞书不被拦截？**
- 飞书域名 `open.feishu.cn` 是国内备案域名
- 微信和飞书都是字节/腾讯系产品，互相没有封锁
- 飞书 Webhook 接口响应极快（< 500ms）

**结果**：
- 浏览器 ✅ → Worker 入库 → 秒级响应
- 微信 ✅ → 直发飞书 → 秒级响应
- 联通网络 ✅ → 直发飞书 → 秒级响应

### 5.4 方案四：JSONP / Image Beacon / 表单提交（未采用）

**思路**：用不受 CORS 限制的方式上报数据。

| 方法 | 问题 |
|------|------|
| JSONP | Worker 不支持，需要额外接口 |
| Image Beacon (`new Image().src = ...`) | 只能 GET，数据量受限 |
| 表单提交到 Worker | Worker 不直接支持表单解析，需要额外处理 |
| **fetch + 飞书兜底** ✅ | 最简单可靠，一条链路两种出口 |

## 六、飞书作为"数据兜底通道"的设计

### 6.1 为什么选飞书做兜底

| 特性 | 飞书机器人 | 其他备选 |
|------|-----------|----------|
| 费用 | 免费 | 企业微信机器人也免费 |
| 微信内可访问 | ✅ | ✅ |
| 消息格式 | 纯文本/Markdown/卡片 | 类似 |
| 群消息可批量导出 | ✅ | 企业微信也支持 |
| 解析难度 | 低（文本正则） | 类似 |
| 管理后台导入 | 已开发 | 需自行开发 |

### 6.2 飞书消息格式设计

为了让后续批量导入方便，消息格式需要结构化：

```
【4D天性测评结果 | 需手动导入】
姓名：张三
主导色彩：绿色·培养型（Cultivating）

四项得分：
绿色（培养型）：30
黄色（包容型）：24
蓝色（展望型）：22
橙色（指导型）：18

决策偏好：
情感(F)：27 | 逻辑(T)：17
直觉(N)：26 | 感觉(S)：18
```

**标签含义**：
- `[已同步]`：Worker 入库成功，数据已在数据库
- `[需手动导入]`：微信/网络异常环境，数据在飞书消息里，需要管理员导入

### 6.3 飞书消息批量导入

管理后台提供"从飞书消息导入"功能：

```javascript
// 解析逻辑：按标题分块 + indexOf 提取字段
function parseImport() {
  var lines = text.split('\n');
  
  // 按标题分块（不按空行，避免同一条消息被劈开）
  var blocks = [];
  var current = null;
  for (var i = 0; i < lines.length; i++) {
    var line = lines[i].trim();
    if (!line) continue;
    if (line.indexOf('【4D天性测评结果') !== -1) {
      if (current) blocks.push(current);
      current = [];
    }
    if (current) current.push(line);
  }
  
  // 逐块提取字段
  for (...) {
    // 用 indexOf 找冒号，不用正则（避免特殊字符问题）
    var colon = line.indexOf('：');
    if (colon === -1) colon = line.indexOf(':');
    if (line.indexOf('姓名') !== -1) item.username = line.substring(colon + 1).trim();
    // ... 同理提取其他字段
  }
}
```

**关键教训**：不要用正则的字符类（如 `[\s\S]`）匹配多行文本，在某些 JS 环境下会失效。**改用 `indexOf` 和字符串分割是最稳健的。**

## 七、常见错误与解决方案

### 7.1 CORS 跨域问题

**现象**：浏览器控制台报错 `CORS policy: No 'Access-Control-Allow-Origin' header`。

**原因**：Worker 返回的响应没有加 CORS 头。

**解决**：
```javascript
const corsHeaders = {
  'Access-Control-Allow-Origin': '*',
  'Access-Control-Allow-Methods': 'GET, POST, OPTIONS',
  'Access-Control-Allow-Headers': 'Content-Type, X-Admin-Key',
};

// 所有响应都要带 CORS 头
function json(data, status = 200) {
  return new Response(JSON.stringify(data), {
    status,
    headers: { ...corsHeaders, 'Content-Type': 'application/json' }
  });
}
```

**特别注意**：Worker 的 `json()` 响应**也必须**带 `Access-Control-Allow-Origin`，否则浏览器会拒绝读取响应体。

### 7.2 StrReplaceFile 工具截断大文件

**现象**：用 `StrReplaceFile` 替换大段代码后，文件被截断到约 457 行，后续代码丢失。

**原因**：工具对大文件替换有长度限制。

**解决**：
- 对大文件（>400 行）的修改，**分段替换**或改用 `WriteFile`/`AppendFile`
- 或者直接用 Git 命令操作文件

### 7.3 Git 操作冲突

**现象**：本地 `git push` 报错 `rejected: non-fast-forward`。

**原因**：在 GitHub 网页上直接编辑了文件，远程有本地没有的提交。

**解决**：
```bash
# 先拉取远程更新，合并后再推送
git pull origin main --no-rebase
git push origin main

# 如果冲突严重，直接强制覆盖（会丢失远程修改）
git push origin main --force
```

**最佳实践**：尽量只在一个地方编辑（本地或网页），不要混用。

### 7.4 GitHub Pages 缓存

**现象**：`git push` 后网页没有更新。

**原因**：GitHub Pages 部署需要 1-3 分钟，浏览器也会缓存静态资源。

**解决**：
- 等 2-3 分钟后刷新
- 用 `?v=2` 后缀绕过缓存（如 `https://xxx.github.io/?v=2`）
- Chrome DevTools → Network → Disable cache

### 7.5 微信 UserAgent 检测不准确

**现象**：加了 `/MicroMessenger/i.test(navigator.userAgent)` 检测，但微信内还是走了 Worker 路径。

**原因**：
- 企业微信的 UA 是 `wxwork` 不是 `MicroMessenger`
- 某些微信版本 UA 格式有变化
- `navigator.userAgent` 在某些内嵌 WebView 中可能不可靠

**解决**（三保险检测）：
```javascript
var ua = navigator.userAgent.toLowerCase();
var isWechat = ua.indexOf('micromessenger') !== -1 
            || ua.indexOf('wxwork') !== -1 
            || typeof WeixinJSBridge !== 'undefined';
```

### 7.6 飞书消息解析失败

**现象**：粘贴飞书消息到导入框，解析不出字段。

**原因**：
- 正则表达式使用了字符类（如 `[\s\S]*?`），在某些 JS 引擎中表现不一致
- 全角冒号 `：`和半角冒号 `:` 混用
- 消息被复制时格式被破坏

**解决**：
- 不用正则匹配多行文本，改用**按标题分块 + 逐行 indexOf 提取**
- 同时兼容全角和半角冒号
- 按 `【4D天性测评结果` 作为分块标志，而不是空行

## 八、最佳实践指南

### 8.1 搭建类似系统的推荐步骤

```
Step 1: 准备前端页面（纯 HTML/CSS/JS）
    ↓
Step 2: 注册 Cloudflare 账号，创建 Worker
    ↓
Step 3: 创建 D1 数据库，设计表结构
    ↓
Step 4: 开发 Worker API（提交/查询/导出）
    ↓
Step 5: 前端接入 API，测试浏览器环境
    ↓
Step 6: 微信内测试，确认是否被拦截
    ↓
Step 7: 如果被拦截，添加微信检测 + 飞书兜底
    ↓
Step 8: 开发管理后台（内嵌在 Worker 中）
    ↓
Step 9: 开发飞书消息批量导入功能
    ↓
Step 10: 全链路测试，上线
```

### 8.2 代码结构建议

```
project/
├── index.html              # 前端页面（GitHub Pages）
├── cloudflare-worker/
│   └── worker.js           # Worker 后端（API + 管理后台）
├── wrangler.toml           # Worker 配置
└── README.md               # 项目说明
```

### 8.3 数据库表结构设计

```sql
CREATE TABLE results (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT,
  f INTEGER, t INTEGER, n INTEGER, s INTEGER,
  green INTEGER, yellow INTEGER, blue INTEGER, orange INTEGER,
  primary_color TEXT, primary_type TEXT,
  source TEXT DEFAULT 'web',  -- 'web' | 'manual'
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

**`source` 字段的重要性**：
- `web`：浏览器正常提交，数据可信度高
- `manual`：从飞书消息导入，数据经过人工确认
- 便于统计不同来源的数据量

### 8.4 安全建议

| 方面 | 措施 |
|------|------|
| API 鉴权 | 管理接口用 Cookie + URL key 双重验证 |
| 数据校验 | Worker 端校验字段类型和长度 |
| 敏感信息 | Worker 密钥用环境变量，不硬编码 |
| 防刷限流 | Worker 免费版有 10 万次/天限制，恶意刷量会触顶 |

### 8.5 监控与兜底

| 监控点 | 方法 |
|--------|------|
| Worker 是否存活 | 访问 `/health` 接口 |
| 数据是否入库 | 管理后台查看最新记录 |
| 微信环境是否正常 | 飞书消息是否秒到 |
| 数据库容量 | D1 免费版 500MB，定期清理或导出归档 |

### 9.1 前端提交逻辑（核心分支）

```javascript

const FEISHU_WEBHOOK = 'https://open.feishu.cn/open-apis/bot/v2/hook/xxx';

function sendFeishu(tag, statusText) {
  var tagText = tag ? ' | ' + tag : '';
  var feishuText = '【测评结果' + tagText + '】\n姓名：' + name + '\n...';
  fetch(FEISHU_WEBHOOK, {
```

```javascript
function checkAdminKey(url, env, request) {
  const key = getAdminKey(url, env, request);
  return key && key === env.ADMIN_KEY;
}

function getAdminKey(url, env, request) {
  // 优先从 URL 参数取
  const urlKey = url.searchParams.get('key');
  if (urlKey) return urlKey;
  // 其次从 Cookie 取
  const cookie = request.headers.get('Cookie') || '';
  const match = cookie.match(/admin_key=([^;]+)/);
  return match ? decodeURIComponent(match[1]) : null;
}
```

## 十二、总结

### 核心认知

1. **微信不是不让你联网，是不让你连某些域名**。`*.workers.dev` 被静默拦截，表现为 TCP 挂起而非直接报错。
2. **运营商也会拦截**。联通对 Cloudflare 默认域名的阻断是彻底的，所以"浏览器环境"也不能 100% 依赖 Worker。
3. **飞书 Webhook 是国内网络环境下最可靠的兜底通道**。响应快、不被拦截、免费。
4. **前端环境检测要用多重保险**。单一 `UserAgent` 匹配不可靠，企业微信、内嵌 WebView 的 UA 各不相同。

### 可复用的模式

这个架构和解决方案可以复用到任何"用户填写 → 数据收集"的场景：
- 在线测评/问卷
- 活动报名
- 信息收集表
- 投票系统
- 预约登记

**只需要替换前端页面和数据库字段，核心链路（GitHub Pages + Worker + D1 + 飞书兜底）完全不变。**
