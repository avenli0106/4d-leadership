# Supabase vs Cloudflare Worker+D1 方案对比

> 背景：两种方案都能实现"零成本数据收集网页"，但在微信环境、网络稳定性、功能扩展性上有明显差异。

---

## 一、为什么 Supabase 没被微信拦截？

### 1.1 域名差异

| 服务 | 默认域名 | 微信拦截状态 |
|------|----------|-------------|
| Cloudflare Worker | `*.workers.dev` | ❌ **被拦截**（TCP 静默挂起）|
| Supabase | `*.supabase.co` | ✅ **目前正常** |

**核心原因**：微信的拦截名单是人工/策略维护的，`workers.dev` 因为被大量开发者用于各种服务（包括一些不合规内容），已经被明确列入限制列表。而 `supabase.co` 主要做数据库/API 服务，目前还没被大规模封锁。

> ⚠️ **但要注意**：这不代表 `supabase.co` 永远不会被拦截。如果未来用的人多，或者出现滥用，同样可能被加入限制列表。

### 1.2 网络路径差异

```
Cloudflare Worker:
  用户 → 微信/运营商 → Cloudflare Edge → Worker 执行 → D1 数据库
                                ↑
                        workers.dev 域名被拦截

Supabase:
  用户 → 微信/运营商 → Supabase CDN/Edge → PostgreSQL API
                                ↑
                        supabase.co 目前正常
```

Supabase 底层也用了 Cloudflare/Fastly 等 CDN，但它的域名不在微信的黑名单里，所以能正常连通。

## 二、方案详细对比

### 2.1 架构复杂度

| 维度 | Supabase | Cloudflare Worker+D1 |
|------|----------|----------------------|
| **是否需要写后端代码** | ❌ 不需要 | ✅ 需要写 Worker.js |
| **前端调用方式** | 直接 `fetch(supabaseURL)` | 直接 `fetch(workerURL)` |
| **数据库操作** | Supabase 自动生成 REST API | 需要在 Worker 里手写 SQL |
| **学习成本** | 低（会用 HTTP 请求即可）| 中（需要了解 Worker 运行时）|
| **部署方式** | 网页配置 + 自动生成 | 命令行 `wrangler deploy` |

**结论**：Supabase 上手更快，前端开发者一个人就能搞定前后端。Cloudflare 需要多写一个 Worker 文件。

### 2.2 免费额度对比

| 资源 | Supabase Free | Cloudflare Free |
|------|---------------|-----------------|
| **数据库容量** | 500 MB | 500 MB（D1）|
| **请求次数** | 无限（但有限流）| Worker: 10万次/天 |
| **带宽/传输** | 2 GB/月 | 无明确限制 |
| **数据库并发** | 30（同时连接数）| 无明确限制 |
| **Edge Function 执行时间** | 无（直接调 API）| 10 ms CPU 时间 |
| **用户认证** | ✅ 内置（无限用户）| ❌ 需自行实现 |
| **实时订阅** | ✅ 内置 | ❌ 需自行实现 |
| **文件存储** | ✅ 1 GB | ❌ R2 需另配 |

**结论**：
- 小项目（日活 < 1000）：两者额度都够用
- 中等项目：Cloudflare Worker 的 10万次/天更 generous
- 需要用户登录/文件上传：Supabase 更省心

### 2.3 微信/国内网络环境

| 场景 | Supabase | Cloudflare Worker |
|------|----------|-------------------|
| **微信内置浏览器** | ✅ 目前正常 | ❌ `workers.dev` 被拦截 |
| **中国联通网络** | ⚠️ 速度较慢（跨洋）| ❌ `workers.dev` 被完全阻断 |
| **中国移动/电信** | ✅ 可用 | ✅ 可用（Worker）|
| **访问延迟** | ⚠️ 100-300ms（海外节点）| ✅ 50-100ms（Edge 节点）|

**关键认知**：
- Supabase 服务器主要在美国/欧洲，国内访问要跨洋，**延迟明显**
- Cloudflare Worker 有国内 Edge 节点（香港/台湾/日本），延迟低得多
- 但 Worker 的域名被拦截，所以这个延迟优势微信用户享受不到

### 2.4 数据库能力

| 特性 | Supabase (PostgreSQL) | Cloudflare D1 (SQLite) |
|------|----------------------|------------------------|
| **SQL 标准** | 完整 PostgreSQL | SQLite（简化版）|
| **复杂查询** | ✅ 支持 | ⚠️ 部分支持 |
| **事务** | ✅ 完整 ACID | ✅ 支持 |
| **关联查询 JOIN** | ✅ 强大 | ⚠️ 基础支持 |
| **全文搜索** | ✅ 内置 | ❌ 不支持 |
| **存储过程/触发器** | ✅ 支持 | ❌ 不支持 |
| **数据导出** | ✅ SQL/CSV | ✅ CSV（管理后台自实现）|

**结论**：Supabase 的数据库能力远强于 D1。如果项目后期需要复杂统计、关联查询、全文搜索，Supabase 更从容。

## 三、两种方案的核心优劣势

### Supabase ✅ 优势
1. **开箱即用**：创建项目就有 REST API，不需要写一行后端代码
2. **微信目前可用**：`supabase.co` 没被拦截，链路简单
3. **功能丰富**：认证、实时、存储、Edge Functions 全套服务
4. **PostgreSQL**：数据库能力强大，扩展性好
5. **社区活跃**：文档完善，生态丰富

### Supabase ⚠️ 劣势
1. **国内访问慢**：服务器在海外，延迟 100-300ms，用户体验受影响
2. **稳定性风险**：`supabase.co` 目前没被拦截，但不保证长期
3. **免费额度较紧**：2GB 带宽/月，如果页面有图片或流量大容易超
4. **数据出海**：数据存储在海外，涉及敏感信息有合规风险
5. **冷启动**：免费版数据库长时间不用会进入休眠，首次请求慢

### Cloudflare Worker+D1 ✅ 优势
1. **全球 Edge 节点**：延迟极低（50-100ms），体验好
2. **免费额度 generous**：10万次请求/天，基本用不完
3. **Worker 灵活**：可以写任意后端逻辑，不只是数据库操作
4. **管理后台一体化**：Worker 可以内嵌 HTML，不需要单独部署
5. **飞书兜底成熟**：即使被拦截，也有完整的数据回收方案

### Cloudflare Worker+D1 ⚠️ 劣势
1. **微信需要兜底**：`workers.dev` 被拦截，必须额外实现飞书/其他兜底
2. **需要写后端代码**：Worker.js 需要维护
3. **D1 功能弱**：SQLite 不如 PostgreSQL，复杂查询受限
4. **调试麻烦**：Worker 本地调试需要 `wrangler dev`

## 四、选型建议

### 选 Supabase 如果你：
- ✅ 希望最快上线，不想写后端代码
- ✅ 项目目前访问量不大，延迟不敏感
- ✅ 需要用户登录、文件上传等高级功能
- ✅ 愿意接受"目前可用但未来可能被拦截"的风险
- ✅ 数据不敏感，可以存放在海外

### 选 Cloudflare Worker+D1 如果你：
- ✅ 追求极致的访问速度（Edge 节点）
- ✅ 可以接受额外写一个 Worker 文件
- ✅ 需要自定义后端逻辑（如管理后台、CSV 导出、飞书通知）
- ✅ 希望有完整的兜底方案（飞书消息），即使被拦截也不丢数据
- ✅ 对数据合规有要求（可以通过 Workers 做数据处理）

## 五、一个折中方案：Supabase + Cloudflare Worker

其实两者可以结合使用：

```
前端页面（GitHub Pages）
    ↓
Cloudflare Worker（API 网关 + 管理后台）
    ↓
Supabase（数据库 + 用户认证）
```

**优势**：
- Worker 提供国内 Edge 节点的低延迟入口
- Worker 可以处理微信检测、飞书兜底等逻辑
- Supabase 提供强大的 PostgreSQL 和认证能力
- Worker 域名被拦截时，Worker 可以转发到 Supabase（或者用自定义域名绕过）

**劣势**：
- 架构变复杂了
- 多一层转发，增加维护成本

## 六、实际建议

**如果你老公已经在用 Supabase 并且跑通了**——那就继续用！方案没有绝对的好坏，能解决问题、稳定运行就是最好的。

**如果未来遇到以下情况，可以考虑迁移或混合**：
1. `supabase.co` 突然在微信里被拦截了 → 参考本项目的飞书兜底方案
2. 用户抱怨加载慢 → 加一层 Cloudflare Worker 做 CDN 加速
3. 免费额度不够用 → Cloudflare 的额度更 generous
4. 需要复杂后台管理功能 → Worker 内嵌管理后台更灵活

**核心原则**：先跑起来，再优化。技术选型永远是"适合当前阶段"而不是"追求最优"。
